# Model Card — Shipment Weight Estimation (v0.1.0-synthetic)

## Overview

Predicts actual packed shipment weight from order/carton features, to replace
the theoretical-weight baseline (sum of catalog item weights + carton tare +
expected packing material).

- **Model type:** Gradient Boosted Trees (`sklearn.ensemble.GradientBoostingRegressor`)
- **Alternatives evaluated:** Linear Regression, Ridge, Random Forest (see `notebooks/01_eda_and_modeling.ipynb`)
- **Status:** Trained and evaluated on **synthetic data only**. No real warehouse data has been received yet.

## Training Data

100% synthetic, generated by `src/shipment_weight/data_gen.py`. The generator simulates:

- 7 item categories with distinct weight/dimension distributions, plus a 3% rate of "unknown" categories never seen at training time.
- A systematic catalog bias per category (theoretical weight is a biased estimate of true item weight).
- Carton selection based on total item volume, with tare weight per carton type.
- Packing material weight as a function of void volume and material type, with packer-to-packer variance (lognormal noise).
- Per-shipment compounding noise that scales with `sqrt(item_count)`, matching the assumption that more items in a carton means more opportunity for weight drift.
- A 2% rate of "mispack" outliers (missed item, extra item, wrong carton) producing large errors.
- A 5% rate of missing catalog weights per item line, flagged via `num_missing_catalog_weights` rather than left as NaN in the aggregate weight.

**This is a stand-in, not real data.** Reported metrics describe how well the
model recovers the structure the generator itself injected — they are useful
for validating the pipeline (features, training, evaluation, serving) end to
end, but they are not evidence of real-world accuracy. Once Medusa's
historical shipment data is available, the generator's assumptions
(noise structure, category mix, outlier rate) should be checked against it
and the model retrained.

## Features

| Feature | Type | Null/unseen handling |
|---|---|---|
| `theoretical_weight_oz` | numeric | median impute |
| `item_count` | numeric | median impute |
| `total_item_volume_in3` | numeric | median impute |
| `num_missing_catalog_weights` | numeric | median impute (0 in practice) |
| `category_avg_weight_error_oz` | numeric | median impute |
| `weight_per_item_oz` (derived) | numeric | median impute |
| `fill_ratio` (derived) | numeric | median impute |
| `num_categories` (derived) | numeric | median impute |
| `carton_type` | categorical | most-frequent impute; one-hot with `handle_unknown="ignore"` |
| `ship_method` | categorical | most-frequent impute; one-hot with `handle_unknown="ignore"` |
| `packing_material` | categorical | most-frequent impute; one-hot with `handle_unknown="ignore"` |
| `category_mode` | categorical | most-frequent impute; one-hot with `handle_unknown="ignore"` |

`handle_unknown="ignore"` means a carton type, ship method, packing material,
or item category never seen during training contributes zero signal from
that feature rather than raising an error — the model still produces a
prediction using the remaining features. See `api/main.py`'s response to an
unseen `MYSTERY_BOX` carton type / `never_seen_category` for a worked
example (no exception, just a less-informed prediction).

## Evaluation (synthetic test set, 8,000 shipments, 80/20 split)

Run `notebooks/01_eda_and_modeling.ipynb` to reproduce. Representative results from a single seed:

| Source | MAE (oz) | RMSE (oz) | Bias (oz) | % within 2oz |
|---|---|---|---|---|
| Theoretical weight baseline | ~10.0 | ~18.1 | ~-2.2 | ~24% |
| Gradient Boosted Trees | ~6.3 | ~10.7 | ~-0.1 | ~35% |

~37% MAE reduction vs. baseline on synthetic data. All four candidate models
(linear, ridge, random forest, GBT) score similarly on this dataset, because
the synthetic noise is largely additive/compounding rather than strongly
nonlinear — this should not be assumed to hold on real data, where
nonlinear interactions (e.g. fragile + multi-item + lightweight carton) may
favor tree-based models more clearly.

**Bias is highest for high item-count shipments and large/XL cartons** — both
of these are where the generator's compounding noise and outlier injection
are concentrated, so this segment-level weakness should be re-checked once
real data is available rather than assumed to transfer.

## Known Failure Modes

- **Mispack outliers (2% of synthetic shipments):** large, effectively
  unpredictable errors (missed/extra item, wrong carton). No feature in this
  dataset signals an upcoming mispack, so the model cannot anticipate these —
  the confidence interval is the main mitigation, not point-prediction
  accuracy.
- **High item-count shipments:** error variance grows with `item_count` by
  construction; the model captures the average trend but individual
  high-count shipments still carry wider uncertainty (see bias-by-item-count
  table in the notebook).
- **Unseen carton types / categories:** degrades to "ignore this feature's
  signal," not a crash — but accuracy for genuinely novel carton types or
  item categories is necessarily worse than for ones seen in training, since
  one-hot encoding can't generalize across categories it never saw.

## Out-of-Distribution Behavior

- Predictions are not clipped or sanity-checked against physical bounds
  (e.g. weight can't be less than carton tare) at the model level — only
  `theoretical_weight_oz` and `total_item_volume_in3` are validated as
  positive at the API layer (`api/schemas.py`). Extremely unusual inputs
  (e.g. `item_count=500`) will still produce a number, not a flagged warning.
- The confidence interval (`±1.645 × residual_std`, a 90% Gaussian band) is a
  single global estimate from held-out synthetic data. It does not widen for
  inputs far from the training distribution — a genuine limitation; a
  conformal-prediction or quantile-regression approach (noted in `README.md`)
  would be a better fit for OOD-aware intervals and should be considered once
  real data volume supports per-segment interval estimation.
